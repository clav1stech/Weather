## Architecture du dashboard (règles)
- **Pipeline ≠ dashboard.** Les fichiers de collecte à la racine (`Forecast.py`, `Forecast_legacy.py`, `validate_cross_pipeline.py`, `run_dual.py`) sont la partie critique : ne jamais les remanier à l'occasion d'un chantier dashboard, et ne pas les déplacer (le workflow, Streamlit Cloud et `config.BASE_DIR` supposent la racine). Les utilitaires hors exploitation (`tools/migrate.py` — one-off historique, `tools/export_project.py`, harnais de non-régression) vivent dans `tools/` ; le profil `export_project.py --outline` reste une vue globale compacte destinée à la réflexion avec un chat IA (architecture/API publique, jamais les corps de fonctions). Le dashboard n'importe du pipeline que `Forecast.persist`/`load_existing`, les helpers xlsx de `validate_cross_pipeline` et les constantes de `run_dual` — ne pas élargir cette surface.
- **Monorepo** : le package `app/` du canicule vit dans `apps/canicule/app/` (exposé sur `sys.path` par `meteo_app.py`, les harnais et `tests/` — les imports internes restent `from app...`). `core/` est **sous** toutes les couches (importable partout côté dashboard) et **config-agnostique** : jamais d'`import config` ni `app.*` dans `core/` — les réglages arrivent en paramètres, liés par les adaptateurs `app/stats/…`, `app/services/…`, `app/ui/theme.py` qui conservent les signatures historiques (contrat des pages et des harnais). Le pipeline racine n'importe jamais `core/` (il garde ses implémentations inline). `apps/snow/` : pipeline/ et app/ opérationnels (entrée `snow_app.py` racine), parquets dans `apps/snow/data/` — jamais dans `data/` (réservé canicule/Paris).
- **Couches à sens unique** : `core` ← `app/runtime` ← `app/data/db` ← `app/stats` ← `app/data/runsets` ← {`app/ui`, `app/pages`, `app/domains`}. Jamais d'import entre pages ni entre domaines.
- **Un phénomène métier = un domaine** `app/domains/<nom>/` (`logic.py`, `charts.py`, `page.py`) enregistré dans `app/domains/__init__.py` (`DOMAIN_PAGES`) — ajouter un domaine ne modifie ni les domaines existants, ni `app/pages/`, ni `meteo_app.py`. Checklist : `docs/CODEMAP.md` § Ajouter un domaine.
- **Non-régression obligatoire** pour tout refactor/factorisation du dashboard : `tools/check_non_regression.py` (calculs) et `tools/ui_snapshot.py` (rendu AppTest des pages), en mode `capture` AVANT modification puis `check` APRÈS — 100 % identique attendu (mêmes données, même heure pleine ; références non versionnées, cf. codemap). Les deux harnais sont en lecture seule sur les données.
- `APP_VERSION` reste dans `meteo_app.py` (`export_project.py` l'y lit par regex). **Versionnage sémantique X.Y.Z** :
  - **Z** : incrémenté à CHAQUE commit contenant une modification, SAUF les petits ajustements purement internes sans impact sur le comportement du dashboard ou du pipeline (ex. `export_project.py`, scripts de tooling, corrections de doc, réglages d'environnement local) — ces ajustements internes ne déclenchent pas de bump. En cas de doute sur si un commit est « interne » ou « avec impact visible », demander avant de trancher plutôt que de bumper par défaut.
  - **Y** : incrémenté UNIQUEMENT sur demande explicite de l'utilisateur, à une étape importante du projet (typiquement un chantier structurant nécessitant une branche dédiée, qui remet Z à 0) — peut être suggéré par Claude, jamais décidé unilatéralement.
  - **X** : jamais décidé par Claude, uniquement sur demande explicite de l'utilisateur, réservé à une nouvelle version majeure.
  Le message de commit du bump commence par `vX.Y.Z – <résumé>` (convention existante de l'historique).
- **Versioning neige SÉPARÉ** : `SNOW_APP_VERSION` dans `snow_app.py`, numérotation et historique totalement indépendants — jamais de bump synchronisé avec `APP_VERSION`, jamais d'entrée neige dans le `CHANGELOG.md` racine. Changelog dédié : `apps/snow/docs/CHANGELOG.md` (entrées `## [X.Y.Z] - date` en tête de fichier, tenu à la main dans le même commit que le bump — `tools/generate_changelog.py` ne couvre que le canicule). Message de commit d'un bump neige : `snow-vX.Y.Z – <résumé>`. **Même sémantique Z/Y/X que le canicule ci-dessus** (Z à chaque commit non-interne, Y et X uniquement sur demande explicite de l'utilisateur — Y peut être suggéré par Claude à une étape importante, jamais décidé unilatéralement). **Seule différence neige : phase développement X=0** — toute la branche dev/snow reste en 0.Y.Z tant que la bascule production n'a pas eu lieu. Le passage à **1.0.0 (bascule production, tag `snow-v1.0`) n'intervient qu'au merge réel en `main`, jamais avant, uniquement sur instruction explicite de l'utilisateur** — comme le X canicule, jamais décidé par Claude. Aucun tag/release publié avant le merge (des tags mobiles internes `snow-v0.Y` sont tolérés, sans promotion Latest).
- **Tags et releases GitHub : un seul tag/release par version Y** (ex. `v2.4`, `v2.5`), pas un par Z. **Préfixes par app depuis 2026-07** : la neige tague `snow-vX.Y` dès sa première release (`snow-v0.1`) ; le canicule bascule sur `heat-vX.Y` à partir de son PROCHAIN bump Y (première occurrence : `heat-v2.7`) — les tags/releases `vX.Y` déjà existants (`v2.6` inclus) restent intouchables tels quels, on ne les renomme jamais ; tant que la ligne 2.6 reçoit des Z, son tag mobile reste `v2.6`. Le tag `vX.Y` est **mobile** : à chaque bump Z sur cette ligne, on le retague (`git tag -f`/suppression+recréation) sur le nouveau commit plutôt que de créer `vX.Y.Z`. Le corps de la release s'enrichit au fil des Z (ne pas écraser l'historique des sections précédentes) et vit dans `docs/releases/<tag>.md`, versionné avec le code. Ne créer un nouveau tag/release que pour un nouveau Y. Depuis un environnement distant sans accès API releases (force-push de tag refusé par le proxy git), le déplacement du tag et la mise à jour de la release passent par le workflow manuel `release-sync` (`workflow_dispatch` : inputs `tag`, `target_sha`, `make_latest`).
- **Passage en "Latest" : uniquement sur demande explicite de l'utilisateur.** Une release mise à jour (nouveau Y ou Z enrichi) reste en l'état (souvent Pre-release) tant que l'utilisateur n'a pas dit de la promouvoir — ne jamais le faire de sa propre initiative, même après un bump Z.

## Invariants à NE JAMAIS casser

### Intégrité des données (règle absolue de non-régression)
- **Tout le projet repose sur ces données** : leur perte est irréversible. Aucune opération ne doit jamais risquer d'altérer, dégrader ou perdre le contenu de `data/database_paris.parquet` ou des xlsx legacy.
- **Données legacy (xlsx Météociel, `legacy/`) : intouchables.** C'est l'assurance-vie du projet, le dernier recours en cas de corruption du parquet. Ne **jamais** les modifier, réécrire, régénérer ni remplacer (y compris par les données d'un autre run) — quelle que soit la raison, même pour « corriger » une incohérence apparente. Lecture seule, toujours.
- **Le parquet ne peut être corrigé que sur demande explicite de l'utilisateur**, suite à un problème identifié et discuté — jamais de correction spontanée, même évidente. Toute correction commence par une **sauvegarde préalable** du fichier (copie datée avant modification).
- **Toute édition du parquet doit prouver sa non-régression** : vérifier avant/après que les données existantes ne sont pas altérées et qu'aucun run correctement récupéré n'est remplacé par des données de moindre qualité (portée réduite, valeurs manquantes, run fantôme). En cas de doute sur l'effet d'une écriture → ne pas écrire, demander.
- **Import ciblé legacy → parquet** (page « Lancer le pipeline », `import_legacy_run`) : ne comble que des **absences avérées** — un couple (run_date, modèle) n'est proposé/importé que s'il n'a **aucune** valeur valide côté Open-Meteo (`legacy_import_candidates`), avec re-contrôle d'absence au moment de l'écriture, sauvegarde datée préalable du parquet, xlsx en lecture seule et fusion via `Forecast.persist`. Ne jamais l'assouplir en outil de « remplacement » de données OM existantes.

### Historique des runs (ne pas écraser)
- **Jamais de perte d'historique.** La fusion (`persist`) se déduplique par couple **(run_date, modèle)**, jamais par `run_date` seul. Un modèle absent d'un fetch **conserve** son run antérieur intact.
- **Écriture atomique obligatoire** : écrire dans un `.tmp` puis `os.replace` — jamais d'écriture directe dans le parquet (pas d'état partiel sur le disque).
- **Valider avant d'écrire** (`_validate`) : ne jamais persister un run vide ou sans valeur valide.

### Détection des runs partiels / fraîcheur
- La distinction « échéances réellement renouvelées » vs « queue recollée de l'ancien cycle » est **empirique et échéance par échéance** (`mask_stale_tail`, seuil `FRESHNESS_EPS`). **Ne pas** la remplacer par une table d'horizons codée en dur ni par une troncature par horizon nominal — l'horizon réel d'un cycle varie d'un jour à l'autre.
- L'identité du cycle (`run_date`) se déduit de la **dernière échéance publiée** (init + horizon), pas de la première (rebouchée par l'API depuis 00:00 local).
- Les garde-fous `RUN_SNAP_TOLERANCE_H` / `RUN_INFER_MAX_SHIFT_H` ne doivent que **corriger vers un cycle voisin** (ex. 12Z→06Z), jamais téléporter. En cas d'ambiguïté → repli sur la détection horloge.
- **Chaque modèle a son propre `run_date`** dans un même fetch (cycles différents). Ne jamais supposer un cycle global partagé.
- **La portée réelle d'un run est CONTIGUË** (`_contiguous_reach_h`, seuil `PERSIST_MAX_GAP_H`) : dernière échéance valide atteignable depuis `run_date` sans trou > 24 h entre échéances valides successives, en ignorant le passé rebouché (`valid_time < run_date`). Ne jamais la remplacer par un simple `max(valid_time) − run_date` : une réponse creuse de l'API (heures rebouchées en tête + point parasite isolé en queue, tout le reste NaN) simulerait une portée pleine, serait persistée comme run fantôme (ligne droite sur les graphes) PUIS bloquerait le vrai run via la garde anti-régression. Cette métrique unique sert aux deux garde-fous ci-dessous.
- **Ne persister un run frais que s'il est complet** (`filter_fresh_rows`/`_meets_persist_horizon` dans Forecast.py) : portée réelle contiguë ≥ `horizon_h − PERSIST_HORIZON_TOLERANCE_H` (ou `MIN_PERSIST_HORIZON_H` si `horizon_h` inconnu, ex. GEM), sinon laissé de côté (ancien run complet conservé, comme un cycle inchangé). Trois causes distinctes, même traitement : un modèle à horizon plein 4×/j (AIFS, GEFS) juste **en cours de calcul** (se résout au poll suivant), un cycle **nativement plus court** par construction (ex. ECMWF ENS à 6Z/18Z ≈ 144 h, jamais 360 h — ce cycle ne sera donc simplement **jamais persisté**, volontairement), ou une **réponse creuse** de l'API (portée contiguë ≈ 0). On continue de checker le modèle à chaque poll (Metadata API + heuristique) ; seule la persistance est conditionnée.
- **Ne jamais régresser un (run_date, modèle) déjà en base** : `persist()` appelle `_drop_regressions` en dernier rempart, indépendant de `filter_fresh_rows` — si `fresh` contient un couple (run_date, modèle) déjà stocké mais avec une portée réelle contiguë INFÉRIEURE à celle déjà en base (glitch API, réponse tronquée à un poll), ce couple est écarté de `fresh` avant fusion : le run existant, plus complet, n'est jamais remplacé par une version régressive. Un `run_date` nouveau pour ce modèle n'est pas concerné (rien à régresser, c'est `filter_fresh_rows` qui juge sa complétude).
- **`Forecast.persist(fresh, existing)` écrit TOUJOURS sur le vrai `C.DB_PATH`**, quels que soient les DataFrames passés en argument (pas de mode dry-run) — ne **jamais** l'appeler avec des données de test/synthétiques, y compris pour un test d'intégration ponctuel : ça écrase la base de production immédiatement (`os.replace`). Pour tester `persist()`, monkeypatcher `config.DB_PATH` vers un fichier temporaire, jamais un appel direct.
- **Variable ajoutée à `config.VARIABLES` après coup (ex. z500) sur un run déjà stocké** : un cycle inchangé (variable historique comme t850 identique) est normalement écarté par `filter_fresh_rows` comme « rien de neuf » — correct pour cette variable, mais bloquerait alors À VIE la persistance de la nouvelle variable sur ce run, qui ne redeviendra jamais « frais ». `complete_missing_vars()` traite ce cas séparément : comble une colonne ENTIÈREMENT absente (jamais une colonne partiellement renseignée) sur le DERNIER run stocké d'un modèle, à partir du poll courant, sans passer par `mask_stale_tail` (une variable absente n'a par définition aucune « queue périmée » à détecter) et sans toucher aux variables déjà renseignées. Ne s'applique que si le poll renvoie exactement le même `run_date` que celui stocké (sinon c'est un cycle réellement nouveau, géré normalement).

### Modèles principaux vs modèles d'appoint
- Les modèles d'appoint (`main: False`, ex. GEM) ne sont **jamais backfillés** et n'existent qu'à leurs cycles réels (`cycles` dans config). Comparaison cycle-à-cycle uniquement.
- Le backfill inter-runs du dashboard (`completed_pooled_sub`) ne concerne que les modèles **principaux**, échéance par échéance, jusqu'à `n-3`. Il ne comble QUE les échéances **à venir** (`valid_time ≥ cycle du run`) : les heures antérieures au cycle (rebouchées par l'API depuis 00:00 local, aussi servies par le run précédent) sont du passé, jetées par la convergence — les backfiller ferait apparaître à tort presque tous les runs comme « complétés » par un ancien (bruit massif en grille horaire OM).
- **`cycles` ≠ `expected_cycles`** : `cycles` liste tous les cycles où un modèle **peut** publier (utilisé par le pipeline pour savoir quand appeler l'API). `expected_cycles` liste les cycles où son absence est une **anomalie à alerter** — ex. ECMWF tourne à 0/6/12/18Z mais n'est *requis complet* qu'à 0Z et 12Z ; son absence à 6Z/18Z n'est pas une anomalie. Les alertes d'interface (`latest_refresh_status`, `page_explore`, `page_convergence`, `_missing_by_run`) utilisent toujours `expected_cycles` (`C.EXPECTED_CYCLES_BY_LABEL`), jamais `cycles`.

### Vues combinées (super-ensemble global)
- Les vues **combinées** (Vue d'ensemble, Indicateur de canicule — **pas** *Explorer un run* ni *Convergence*) poolent, pour **chaque modèle, son dernier run à HORIZON PLEIN** (`latest_complete_run_sub`), chacun gardant son propre cycle. La complétude se mesure **empiriquement** sur la portée réelle du run stocké (`max valid_time − run_date ≥ horizon_h − FULL_HORIZON_TOLERANCE_H`) — **jamais** par une règle codée en dur sur l'heure de cycle : un 6Z/18Z réellement long est éligible, un 0Z/12Z anormalement court est écarté. Modèle sans `horizon_h` (GEM) → dernier run non vide ; aucun run à horizon plein → repli sur le dernier non vide, signalé « horizon réduit ».
- La Vue d'ensemble peut être **rejouée à un cycle antérieur** (sélecteur « Vu depuis », param `as_of` de `latest_complete_run_sub`) : base filtrée `run_date ≤ cycle` puis même logique de sélection — la carte Tendance se compare toujours au jeu précédent **relatif à la version affichée**. La référence « présent » des KPI est alors le cycle choisi, jamais l'horloge.
- Exception voulue sur la page canicule : ses sections vulgarisées « évolution au fil des runs » et « confiance » (`trend_daily_medians`) réutilisent la **mécanique de la page Convergence** (super-ensembles complétés `completed_super_ensemble_daily`, filtre `_convergence_runs`), pas `latest_complete_run_sub` — comparer des runs entre eux exige des pools à modèles équivalents (backfill), ce que la sélection « dernier run complet » ne garantit pas. Ne pas « unifier » les deux.
- Les **KPI de la Vue d'ensemble** sont config-driven (`KPI_*` dans config.py) et calculés sur les **échéances à venir uniquement** (les heures passées rebouchées par l'API fausseraient prochaine échéance, pic, tendance, anomalie). Le « jour à risque » mêle **probabilité × sévérité** : proba journalière ≥ `KPI_RISK_PROB_MIN` OU dépassement attendu E[max(T − seuil, 0)] ≥ `KPI_RISK_EXCESS_MIN_C` (colonne `exces` de `daily_risk`) — ne pas le réduire à un seuil de proba seul, le second critère capte les queues chaudes à proba modeste.

### Explorer un run : « Dernier run » et tableaux d'export
- L'option « Dernier run » du sélecteur (`latest_run_sub`) poole le **dernier run non vide de chaque modèle, quel que soit son cycle et sans exigence d'horizon plein** — c'est voulu (fraîcheur maximale, même partielle) ; ne pas la confondre avec `latest_complete_run_sub` (vues combinées, horizon plein requis).
- Les tableaux de l'onglet 🧾 sont **volontairement larges** (destinés à l'export pour analyse par IA) : stats du super-ensemble + par modèle médiane, contrôle (member 0), nb de membres, Δ médiane vs run précédent — ne pas les « alléger » pour la lisibilité écran.
- Le « run précédent » des colonnes Δ (`previous_runs_sub`) se calcule **par modèle** (dernier run strictement antérieur de CE modèle), jamais via un cycle global partagé.

### Variables de contexte (Z500)
- **`t850` (PRIMARY_VAR) reste l'UNIQUE variable pilotant** la détection canicule, les seuils, la sélection/complétude des runs et les KPI. `z500` (géopotentiel 500 hPa — nom API `geopotential_height_500hPa`, le nom `geopotential_500hPa` est invalide) n'est qu'une **couche de contexte synoptique** : médiane d'ensemble seule dans les vues techniques (jamais de spaghetti Z500), signal **qualitatif** côté grand public (anomalie vs normale cosinus `CLIM_Z500_*`, paliers/persistance dans `heatwave/logic.py` — jamais de valeur brute) — en appui du message T850, jamais un critère de risque concurrent.
- **L'absence de z500 est un cas NORMAL, jamais une anomalie** : lignes antérieures à son ajout, runs importés du legacy (Météociel ne publie pas Z500 — ne jamais l'ajouter au scrape ni au contrôle croisé). Tout accès Z500 se dégrade en silence (`None` → rien d'affiché). Côté pipeline : `load_existing` réaligne le parquet historique sur le schéma courant (colonne manquante → NaN), et les mesures de portée/anti-régression comptent une ligne valide dès qu'**une** variable l'est (`dropna(subset=VAR_COLS, how="all")`) — ne pas resserrer, sinon les runs historiques paraîtraient vides et la garde anti-régression serait neutralisée.
- **Vue d'ensemble et Indicateur de canicule affichent Z500 via un pool DÉDIÉ** (`latest_z500_sub` dans `app/data/runsets.py`), pas via le `sub` T850 de `latest_complete_run_sub` : pour chaque modèle, on retient son dernier run contenant une valeur z500 valide, quitte à remonter à un run plus ancien que celui affiché pour T850 (ex. run T850 tout juste rafraîchi sans z500 encore collecté, ou trou ponctuel). Z500 n'ayant ni exigence d'horizon plein ni de complétude (simple contexte), ce pool ne suit pas la politique horizon-plein de `latest_complete_run_sub`. Pool vide (z500 absent de toute la base à la date considérée) → dégradation silencieuse identique à avant (rien affiché).

### Flux annexe Tx/Tn haute résolution (T2m)
- **Flux séparé de bout en bout** : `forecast_t2m_hd.py` (API Forecast standard, PAS l'API Ensemble) → `data/database_paris_t2m.parquet` (`C.T2M_SCHEMA` : fetched_at/model/target_date/tx/tn, append-only, valeurs `daily` de l'API stockées telles quelles, aucun recalcul) → `app/data/t2m.py` (lecture seule). Ne jamais le mélanger à `database_paris.parquet` (pas de `member`, pas de cycle synoptique — les modèles « seamless » n'ont pas de run unique, le flux est daté par instant de collecte `fetched_at`).
- **Horizon 7 jours, un seul couple Tx/Tn affiché par jour** (`T2M_FORECAST_DAYS`) : Météo-France ne publie que J à J+3 (null au-delà, constaté empiriquement), DWD ICON couvre les 7 jours. L'ordre de `T2M_MODELS` = priorité d'affichage (MF tant qu'il existe, ICON en secours) ; jamais deux valeurs pour un même jour, jamais de moyenne/arbitrage entre modèles (choix : cet outil met en forme des prévisions tierces, il n'arbitre pas). Ne PAS combler Tx/Tn avec les modèles d'ensemble (T850) : ce sont des flux et des variables distincts.
- **Indicateur d'incertitude à deux régimes** (`incertitude_txtn` dans `heatwave/logic.py`, seuils `T2M_ECART_*`) : le second modèle HD n'est pas affiché mais sert de recoupement. Les deux présents (J→J+3) → l'écart |MF−ICON| qualifie la fiabilité ; MF absent (J+4→J+6, ICON seul) → valeur indicative « source unique ». Un glyphe `≈` marque en case les jours peu fiables (source unique OU forte divergence) ; le détail se lit au survol. La couleur des cases reste réservée au risque T850 (jamais surchargée par la confiance Tx/Tn).
- **Affichage seul, jamais un critère** : Tx/Tn n'influence ni la détection canicule, ni les seuils, ni la sélection des runs, ni les KPI — texte d'appui dans les cases du calendrier du risque, dont la couleur reste pilotée uniquement par la probabilité T850.
- **Absence = cas normal** : parquet T2m manquant/vide/corrompu/partiel → dégradation silencieuse (`app/data/t2m.py` renvoie un DataFrame vide, la page s'affiche strictement comme sans ce flux). Aucune alerte, aucun comblement.
- **Push git partagé** : le job workflow `fetch-t2m-hd` commit un fichier distinct mais partage `concurrency.group: weather-data-push` — la contention est sur le push git du repo (son cron 2h croise `fetch-api`), pas sur les fichiers.

### Flux annexe observations Météo-France (DPObs)
- **Sécurité de la clé API — règle absolue** : la clé DPObs vit UNIQUEMENT dans le secret GitHub Actions `METEOFRANCE_API_KEY` (CI), dans `.env` gitignoré (local, parsé par `fetch_observations.py` sans dépendance externe), et dans les Secrets Streamlit Cloud sous le même nom (lu via `st.secrets`, cf. `app/services/live_observations.py` — bouton « aperçu instantané » de la page Observations, appel EN DIRECT jamais persisté). Jamais en dur dans le code/commits/logs, jamais loguée même partiellement, jamais de valeur par défaut ni de repli silencieux : variable/secret absent → échec explicite (le dashboard dégrade en message d'indisponibilité, jamais un crash).
- **Flux séparé de bout en bout** : `fetch_observations.py` (API **DPPaquetObs/v2** — la clé couvre v2, PAS v1 qui répond 403 —, header `apikey`, **un seul appel** `/paquet/horaire?id-departement=75` renvoyant toutes les stations du département sur une fenêtre glissante de plusieurs jours) → `data/database_paris_observations.parquet` (`C.OBS_SCHEMA`, append-only) → `app/data/observations.py` (lecture seule) → domaine `app/domains/observations/`. Ne jamais le mélanger aux parquets Ensemble ou T2m. Le paquet département est préféré au mono-station DPObs `/station/horaire` (une obs par appel) : backfill naturel de l'historique et **rattrapage automatique d'un cron manqué** (toutes les heures intermédiaires sont dans le paquet suivant) — ne pas revenir au mono-station. Les stations du département hors `OBS_STATIONS` sont ignorées silencieusement ; une station absente du paquet = panne partielle (les autres persistées, le paquet suivant comble rétroactivement) ; seul l'échec de l'appel département entier est une panne totale (sortie en erreur, parquet intact).
- **Clé de déduplication : (station_id, valid_time)** où `valid_time` = `validity_time` de l'API (heure de l'observation, UTC). **Jamais `reference_time`** : c'est l'heure de production du lot côté Météo-France, identique pour toutes les stations et renouvelée à chaque poll pour une même obs — en faire la clé dupliquerait chaque observation à chaque poll. Une observation stockée n'est **jamais remplacée ni modifiée** (fait acquis, pas une prévision révisable).
- **Unités converties AU PARSING, avant stockage** : t/td/tx/tn en Kelvin → °C, pres/pmer en Pa → hPa (constaté en conditions réelles) — le parquet est directement comparable aux données Open-Meteo. rr1 (mm), u (%), ff (m/s), dd (°), ray_glo01 (J/m²) stockés tels quels.
- **Disponibilité inégale STRUCTURELLE des variables** : les stations du réseau ETENDU (Lariboisière, Luxembourg) ne publient que t/tx/tn/rr1 — humidité/vent/pression y sont null par niveau d'instrumentation, pas par panne ; la pression n'existe qu'à Montsouris (Longchamp RADOME ne la publie pas). Absence → NaN, jamais une valeur inventée ni une alerte. Humidité/vent/pluie/pression ne s'affichent que pour la station de référence (`reference: True`) ; la **comparaison inter-stations ne porte que sur la température**, seule variable mesurée partout.
- **Groupes ICU explicites** : le champ `icu` (`"urbain"`/`"aere"`/`"neutre"`) de `OBS_STATIONS` pilote l'écart ICU — ne jamais le déduire du réseau (la coïncidence ETENDU=urbain / RADOME=aéré est propre à ces 4 stations). Montsouris est `"neutre"` : malgré son profil de parc, elle reste en pratique proche de l'urbain la nuit (constaté empiriquement) et fausserait le groupe aéré (Longchamp seul) — elle est donc exclue du calcul d'écart tout en restant affichée sur les graphiques. Périmètre fermé : exactement ces 4 stations, pas de sélecteur.
- **Panne partielle = cas normal** : une station absente/raréfiée dans le paquet est simplement moins alimentée à ce poll (les autres persistées, rattrapage rétroactif au paquet suivant) ; échec de l'appel département entier → erreur propre, parquet intact. Côté dashboard : parquet absent/vide/corrompu → dégradation silencieuse (DataFrame vide, message d'attente) ; une station muette n'empêche jamais l'affichage des autres ; obs plus vieille que `OBS_PERIMEE_H` → signalée « ancienne », jamais présentée comme actuelle.
- **Comparaison prévu/observé** : uniquement Tx/Tn HD (prévision **unique** pour Paris, cf. `LATITUDE`/`LONGITUDE`) vs Tx/Tn observés par station, sur jours **révolus quasi complets** (`JOUR_COMPLET_MIN_H` : Tn/Tx d'une journée trouée seraient faussés) — l'écart par station est le signal d'ICU local, documenté comme tel dans l'UI. **Jamais T850 vs observations de surface** (variables physiques différentes). Tx/Tn observés journaliers = max des `tx` horaires / min des `tn` horaires (extrêmes intra-horaires de l'API).
- **Affichage seul, jamais un critère** : les observations n'influencent ni la détection canicule, ni les seuils, ni la sélection des runs, ni les KPI.
- **Cadence indépendante** : cron horaire dédié (`20 * * * *`), volontairement découplé du rythme 2 h des autres jobs ; `concurrency.group: weather-data-push` partagé (contention sur le push git, pas sur les fichiers).

### Flux annexe prévision Montsouris « vintages » 15 min (minutely_15)
- **Quatrième flux, TROISIÈME granularité temporelle** : `fetch_montsouris_vintages.py` (API Forecast standard `/v1/forecast`, paramètre `minutely_15`, PAS l'API Ensemble) → `data/database_paris_montsouris_vintages.parquet` (`C.VINTAGE_SCHEMA`), lu en seule lecture par `app/data/vintages.py`. Température/humidité/précip au pas de **15 min sur ~48 h de futur**, **au point de Montsouris** (`C.VINTAGE_LAT`/`VINTAGE_LON`, station de référence — PAS le point Paris générique `LATITUDE`/`LONGITUDE` : ces prévisions seront confrontées aux observations de CETTE station). À ne JAMAIS fusionner avec l'ensemble T850 (horaire, 16 j) ni le Tx/Tn HD (journalier, 7 j) : trois granularités, trois usages. Un seul modèle `meteofrance_seamless` (pas de secours DWD ICON ici — le besoin est la finesse locale court terme, pas un arbitrage). Unités déjà exploitables (°C, %, mm) : **aucun recalcul au parsing** (à la différence des observations MF en Kelvin/Pa).
- **Historique des révisions (VINTAGES) — clé `(valid_time, fetched_at)`, APPEND-ONLY.** C'est l'inversion de l'ancien flux « instant » (upsert sur `validtime` seul, une valeur par échéance) : une prévision est **RÉVISABLE**, sa valeur pour une même échéance change légitimement d'un poll à l'autre — on **conserve** ces révisions successives (chaque poll fige un « vintage »), jamais d'upsert qui écraserait un vintage antérieur. C'est ce qui permet à un futur graphique de comparer, pour une échéance, les prévisions émises il y a 6/12/18/24 h et de visualiser la convergence à l'approche de la réalisation. `persist()` append les vintages frais (dédup défensive sur le couple) puis compacte — jamais d'écrasement.
- **Compaction à deux régimes (`compact`, fonction pure testable) — borne la table.** Fenêtre `± VINTAGE_RETENTION_H` (48 h) autour de l'instant du run : une échéance (`valid_time`) **dans** la fenêtre (passé ou futur) garde **tous** ses vintages ; une échéance **passée hors** fenêtre ne garde que le vintage dont `fetched_at` est le **plus proche** du `valid_time` (dernière estimation avant réalisation — privilégie un vintage live proche plutôt qu'un bootstrap lointain) ; le futur hors fenêtre est laissé tel quel (cas quasi vide, le modèle dépasse rarement 48 h). Ne JAMAIS compacter une échéance encore, même partiellement, dans la fenêtre. Compaction appliquée APRÈS l'append (ne touche que le passé hors-fenêtre → aucune perte possible).
- **Colonne `source` (`"live"`/`"bootstrap"`)** : marque le lot de chaque poll. `bootstrap` = comblement initial via `past_days` (approximation proche de l'observé, **PAS un vrai vintage figé**) ; `live` = poll régulier. Ne pas mélanger les deux sans distinction dans un futur affichage.
- **Backfill via `past_days` adaptatif** : premier run (parquet absent) → `VINTAGE_BACKFILL_PAST_DAYS_INIT` (7 j, amorçage `source="bootstrap"`) ; runs suivants → `VINTAGE_BACKFILL_PAST_DAYS` (1 j, rattrapage d'un trou de cron, `source="live"`). La fenêtre future ~48 h est renvoyée quoi qu'il arrive.
- **Affichage seul, jamais un critère** : n'influence ni la détection canicule, ni les seuils, ni la sélection des runs, ni les KPI. **Affichage branché** : section « Convergence de la prévision — Montsouris » de la page Observations — graphique confrontant la courbe observée (6 min) aux prévisions émises à divers reculs (`vintage_comparison_series`/`vintage_comparison_chart`), complété d'un tableau d'écarts prévision − observé (`tableau_ecarts_convergence`), le tout en lecture seule via `app/data/vintages.py`. NaN en début de fenêtre (donnée non consolidée) = cas normal ; réponse vide/inattendue → DataFrame vide, jamais de crash.
- **Cadence indépendante** : cron `50 * * * *` (toutes les heures — une cadence horaire suffit à capturer la convergence, le modèle seamless ne se réactualisant pas au pas de 15 min de la donnée) ; créneau `:50` libre (évite `:05`/`:10`/`:15`/`:20`/`:25`/`:35`/`:40`/`:45`/`:55`) ; `concurrency.group: weather-data-push` partagé.
- **Ancien flux « instant » retiré** : `fetch_instant.py` supprimé, section `INSTANT_*` de `config.py` remplacée par `VINTAGE_*`. Le parquet historique `data/database_paris_instant.parquet` est laissé en place mais **orphelin** (plus aucun code ne le lit) — à nettoyer plus tard.

Seuil de complétude 336−24 = 312 h : cycles pleins persistés, cycles nativement courts (ECMWF 6Z/18Z) exclus — voulu.
- **Trous STRUCTURELS constatés → NaN, jamais une panne ni un comblement** : AIFS sans iso0/snow_depth/rafales ; flux mean sans iso0 ; GEFS_MEAN sans niveaux de pression (t850/z500/t500/épaisseur NaN) ; AROME France HD sans variables neige (t2m/rafales seulement — ICON-D2 porte seul la neige en maille fine). Épaisseur 1000-500 dérivée AU PARSING (z500 − z1000, z1000 non stocké), jamais sur les lignes spread (l'écart-type d'une différence ne se déduit pas des écarts-types).
- **CI : UN job `fetch-snow`** (cron 2 h à :35, `concurrency.group: weather-data-push` partagé) enchaîne les deux scripts pour limiter les push git, mais chaque flux reste relançable isolément (`workflow_dispatch` : snow / snow-ensemble / snow-hd). Boucle de retry sans `git pull --rebase` (parquets binaires), comme les autres flux persist()/dédup.
- **Extension Météo-France PNT dans ce MÊME job neige 2 h** : ne jamais créer un cron horaire AROME-PI. À chaque passage H+35, le catalogue dynamique pilote une collecte idempotente du dernier run AROME-PI disponible ; AROME-IFS et PE-AROME ne sont téléchargés que sur nouveau cycle 6 h, PE-ARPEGE sur nouveau cycle complet 00/12Z. Un catalogue inchangé produit un no-op sans écriture ni commit. Chaque API utilise exclusivement son secret dédié (`METEOFRANCE_AROME_PI_KEY`, `METEOFRANCE_AROME_KEY`, `METEOFRANCE_AROME_PE_KEY`, `METEOFRANCE_ARPEGE_PE_KEY`), jamais une autre clé en repli silencieux. L'API AROME directe ne collecte qu'AROME-IFS ; AROME France standard reste obtenu via Open-Meteo afin d'éviter un doublon de stockage et de pondération.
- **PE-ARPEGE WCS** : utiliser exclusivement le service Europe 0,1° et une petite emprise autour de Megève, dont les bornes sont alignées à 0,1° exactement. Ce WCS est sensible à l'ordre des axes : `subset=lat(...)` doit précéder `subset=long(...)` (l'ordre inverse ou des bornes non alignées renvoient HTTP 400 selon le produit). Le backend pouvant aussi refuser transitoirement un appel spatial strictement identique, seuls ces HTTP 400 sont retentés de façon bornée ; un refus persistant échoue. Une borne stricte sur la taille de chaque GRIB doit empêcher tout téléchargement silencieux de la grille Europe complète. Le contrôle pouvant précéder les perturbations, ne collecter que le dernier cycle 00/12Z commun au membre 0 et à une perturbation sentinelle.
- **Granularité WCS Météo-France** : un `GetCoverage` PE ne renvoie qu'un champ 2D et impose une unique valeur `time` ; ne jamais tenter de reconstruire 25×51 échéances horaires par avalanche de requêtes. AROME-PI/IFS portent le détail horaire local ; PE-AROME est collecté par membre sur H24/H48, PE-ARPEGE sur les seuls cycles complets 00/12Z à H24/H48/H72/H96, exclusivement total+neige P1D et cellule village. Aucun paramètre non affiché n'est téléchargé. Leurs requêtes sont espacées sous le quota ; les diagnostics de masse d'air restent issus des ensembles globaux existants. Le pipeline ne stocke jamais les GRIB bruts : ecCodes les décode directement depuis les octets et seules les cellules utiles sont conservées.
- **Stockage Météo-France ciblé** : brut normalisé local (PI/IFS/PE-AROME) dans `db_megeve_mf_local.parquet`, brut PE-ARPEGE dans `db_megeve_arpege_pe.parquet`, tous deux en HOT 45 j + COLD append-only. En parallèle, `db_megeve_mf_summary.parquet` conserve sans limite une synthèse compacte par cycle/modèle/site/échéance (moyenne des valeurs, nombre de membres ; mode pour `ptype`, jamais moyenne de codes). Le brut reste la source de vérité pour tout recalibrage ; un run inchangé ne réécrit pas sa synthèse.
- **Dashboard neige (`snow_app.py` racine + `apps/snow/app/`)** : ARCHITECTURE MULTI-SIGNAUX, pas de PRIMARY_VAR unique — les pivots se hiérarchisent PAR ÉCHÉANCE (`snow_config.HORIZON_REGIMES`) : H0-H+48 détail HD horaire sec/pluie/neige aux quatre altitudes + bilan quotidien cm/mm (LPN/iso 0 utilisés au calcul mais jamais tracés sur l'axe des altitudes), classification catégorielle des membres sur J0-J+15, puis pmsl comme TIMING seul à longue échéance (bascule si chute ≥ 5 hPa/24 h, jamais un critère neige). Seuils validés : LPN = iso0 − 300 m (`LPN_MARGE_M`) ; t850 ≤ +1 °C (sommet) / −1 °C (village) = neige probable, ≥ +3 °C = redoux ; KPI « jour à neige » proba×sévérité (proba(cumul ≥ 1 cm) ≥ 0,5 OU attendu ≥ 5 cm, paliers 1/5/20 cm). `domains/neige/weather_type.py` est le point de calibration unique, **à affiner in situ au fil de la saison**, pour le seuil de bruit precip, le ratio neige/précipitation, les pondérations, l'épaisseur et la coupe verticale ; la marge LPN HD reste provisoirement partagée avec l'ensemble et doit être recalibrée en fin de saison. La classification par membre est strictement au village (seul site ensemble portant precip+t850+épaisseur) : sous 0,5 mm/j le membre est sec, de 0,5 à moins de 2 mm/j une trace chaude reste mixte/incertaine et « pluvieux » exige au moins 2 mm/j ; une masse d'air assez froide peut toujours qualifier la neige dès le seuil de précipitation franchi. Pour le TYPE DE TEMPS local, on calcule d'abord une proportion par modèle puis on pondère les familles selon l'horizon : GEFS, global et peu représentatif de l'orographie de la vallée de l'Arve, reste secondaire face à ECMWF/AIFS à courte échéance ; cette minoration ne s'applique jamais aux diagnostics de masse d'air t850/épaisseur, où GEFS reste pertinent et reprend du poids avec l'horizon. t850 domine le score froid (70 %) et l'épaisseur profonde le module (30 %) sans jamais constituer un veto absolu ; les profils contradictoires restent mixtes. Une precip absente reste explicitement non classée, jamais assimilée à sec, et aucune classification sommet silencieuse n'est permise. La Vue d'ensemble conserve toutes les sorties brutes mais ne trace le graphique neige historique que si un signal d'affichage plus sensible que le KPI est atteint (`DISPLAY_NEIGE_*` ou P90 ≥ 1 cm) : quelques traces de membres isolés dans un scénario doux sont expliquées, jamais présentées comme un épisode. z500/t500 : contexte pur, jamais de seuil. L'absence structurelle d'iso0 dans `_MEAN` reste un cas normal, sans repli sur les membres à rétention courte. ## Conventions de travail
- **Config-driven** : tout réglage variable (modèle, variable, seuil, climato) se déclare dans `config.py` — une ligne suffit, sans toucher la logique de parsing/stockage/affichage.
- **Préserver la densité de commentaires** : le code est richement documenté en français (le *pourquoi*, pas seulement le *quoi*). Maintenir ce niveau, et expliquer les invariants subtils.
- **Nature du commentaire, pas son histoire** : un commentaire doit être bref, concis et complet — il éclaire la nature du code (particularités, pièges, invariants, points d'attention), jamais sa genèse. Proscrire les commentaires « de circonstance » qui répondent à une discussion ou à un bug découvert avec l'IA (ex. « corrigé suite à... », « ajouté car Claude a détecté... ») : reformuler en constat intemporel sur le code lui-même.
- **Schéma parquet stable** : ne pas casser la compatibilité de `C.SCHEMA` (`load_db` filtre déjà les modèles legacy orphelins).
- **Contrôle croisé legacy** : médiane-vs-médiane (pas det-vs-det), limité aux runs 0Z/12Z **automatiques** (cf. `validate_cross_pipeline.py`). **Météociel ne publie QUE 0Z/12Z en intégralité et pour les 3 modèles** — les cycles 6Z/18Z d'Open-Meteo n'ont légitimement aucun équivalent legacy à cette cadence (ne jamais les traiter comme une absence anormale).
- **Legacy 6Z/18Z : manuel only, AIFS/GEFS seulement.** `Forecast_legacy.py`/`validate_cross_pipeline.py` acceptent aussi `6Z`/`18Z` en argument (`RUN_HEURES`), mais uniquement via déclenchement manuel (`workflow_dispatch`, choix `legacy_run`) — jamais de cron automatique dessus. À ces cycles, ECMWF ENS est exclu du scrape (non publié en intégralité par Météociel à 6Z/18Z, cf. `expected_cycles`) ; seuls AIFS/GEFS (qui tournent réellement 4×/j) sont récupérés. Le rattrapage inter-cycle (`tenter_rattrapage`) reste limité au couple 0Z/12Z, pas d'équivalent pour 6Z/18Z.
- **Workflow GitHub Actions (`run_forecast.yml`)** : deux jobs à cadences distinctes, pas un seul cron couplé. `fetch-api` (Forecast.py) tourne toutes les 2h — c'est la source principale. `scrape-legacy` (Forecast_legacy.py) ne tourne **automatiquement** qu'aux créneaux réels 0Z/12Z de Météociel (~midi/minuit heure de Paris) et enchaîne toujours avec le contrôle croisé (`validate_cross_pipeline.py`), car Météociel est **structurellement** la dernière source à finaliser un run 0Z/12Z (publication complète décalée de plusieurs heures) — le contrôle croisé n'a donc jamais de raison d'être déclenché ailleurs automatiquement. Le déclenchement manuel (`workflow_dispatch`, `target=legacy`) permet en plus de choisir 6Z/18Z, indépendamment de l'heure courante. `run_dual.py` n'est plus appelé par le workflow ; il ne sert plus qu'au lancement manuel (bouton dashboard). Les deux jobs partagent un `concurrency.group` pour sérialiser leurs push git (évite un conflit quand les deux tombent au même horaire, ex. 10Z/22Z). **Retry sur le push** : `cancel-in-progress: false` protège un job déjà EN COURS, mais pas un job encore EN ATTENTE dans la file — GitHub Actions annule les requêtes en attente les plus anciennes dès qu'une requête plus récente arrive pour le même groupe (ex. déclenchement manuel `target=all` mettant les 6 jobs en file, puis un cron indépendant se déclenche pendant l'attente). Un parquet est **binaire** : un `git pull --rebase` ne sait pas fusionner deux écritures concurrentes du même fichier (conflit non auto-résoluble, contrairement à du texte) — risque déjà rencontré en pratique malgré le `concurrency.group` partagé, en particulier sur le job à cadence la plus rapprochée (`fetch-observations-6m`, 15 min). **4 des 6 jobs** (`fetch-api`, `fetch-t2m-hd`, `fetch-observations`, `fetch-observations-6m`, `fetch-vintages` — les flux `persist()`/dédup par clé) ne font donc jamais de `git pull --rebase` sur leur parquet : leur boucle de retry (5 tentatives) relance tout le cycle collecte+push — en cas d'échec de push, `git reset --hard origin/<branche>` puis re-exécution du script Python, dont `persist()` relit toujours le disque et recalcule la fusion (dédup par clé) sur la base distante à jour, jamais via git. `scrape-legacy` reste sur `until git pull --rebase ... && git push` (texte + xlsx neufs par (date, cycle), conflit binaire seulement si deux runs ciblent le même (date, cycle)) car `validate_cross_pipeline.py` **append** une ligne à `data/cross_check_log.csv` à chaque appel — rejouer le scrape en boucle dupliquerait cette ligne, ce n'est pas idempotent comme les autres flux.
- **Bascule pipeline (`config.PIPELINE_LIVE_SINCE`)** : avant cette date, la base Open-Meteo est **rétro-remplie depuis les xlsx Météociel** (`tools/migrate.py`, one-off historique) → toute comparaison OM↔legacy y est circulaire, et GEM / les cycles 6Z-18Z n'y existent pas. Ne confronter les deux sources qu'à partir de cette date. La détection d'absence d'un modèle se cale, elle, sur sa **1re apparition réelle** dans la base (pas de date en dur par modèle).
- **Git / fichiers** : ne pas committer sans demande explicite. `Export/`, `.venv/`, `__pycache__/` ne sont jamais versionnés.
- **Changelog** : chaque commit qui bump `APP_VERSION` (`vX.Y.Z – résumé`) doit mettre à jour `CHANGELOG.md` **dans le même commit** — une entrée `## [X.Y.Z] - date` reprenant le résumé du commit, ajoutée en tête de fichier. Pas de saisie manuelle séparée : `CHANGELOG.md` se déduit des messages de commit, `tools/generate_changelog.py` régénère/complète le fichier automatiquement (idempotent — n'ajoute que les versions absentes, sans toucher aux entrées déjà écrites). Relancer ce script avant un bump **Y** (nouvelle release) ou après un merge de branche apportant plusieurs commits de version d'un coup, pour vérifier qu'aucune entrée n'a été oubliée.
- **Travail en branche vs `main`** : `main` pour un changement ponctuel à risque quasi nul, immédiatement déployable (bump Z) ; une branche dédiée pour un chantier structurant multi-commits (bump Y). Une branche de travail ne doit **jamais committer de modification sur les `data/*.parquet` existants** (les jobs CI ne poussent que sur `main` toutes les 2h ; tant qu'une branche n'y touche pas, ces fichiers restent identiques entre le point de fork et `main`, donc aucun conflit possible au merge/rebase — git prend automatiquement la version `main`). Pour retravailler avec une donnée fraîche en local pendant le chantier sans risquer de la committer : script `tools/refresh_data_from_main.sh` (à créer si absent) qui écrase localement les fichiers connus depuis `origin/main` et les marque `skip-worktree` (invisibles pour `git status`/`git add`). **Si la branche introduit un NOUVEAU parquet** (nouveau flux/domaine), ce fichier ne pose aucun conflit (ajout pur au merge) mais n'est pas concerné par ce mécanisme avant le merge ; une fois mergé sur `main`, penser à l'ajouter à la liste des fichiers du script pour que les futures branches puissent aussi le rafraîchir.
